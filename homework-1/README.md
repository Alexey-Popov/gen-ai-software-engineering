## 📋 Project Overview

A minimal REST API for banking transactions, built with **Python + FastAPI** and in-memory storage (no database). It implements transaction creation/listing/lookup, account balance and summary calculation, request validation, history filtering, and per-IP rate limiting.

### Features implemented

- **Task 1 — Core API**: `POST /transactions`, `GET /transactions`, `GET /transactions/:id`, `GET /accounts/:accountId/balance`.
- **Task 2 — Validation**: positive amounts with at most 2 decimal places, `ACC-XXXXX` account format, ISO 4217 currency whitelist. Errors are returned as `{"error": "Validation failed", "details": [...]}`.
- **Task 3 — History filtering**: `GET /transactions` supports `accountId`, `type`, and `from`/`to` date-range filters, combinable.
- **Task 4 — Extras**:
  - **Transaction Summary**: `GET /accounts/:accountId/summary`
  - **Rate limiting**: 100 requests/minute per client IP, `429 Too Many Requests` when exceeded.

### Architecture

```
src/
├── main.py             FastAPI app, middleware, exception handlers
├── models.py            Pydantic models (Transaction, enums)
├── validators.py        Field-level validation (amount, account, currency, type)
├── storage.py            In-memory TransactionStore (add/list/balance/summary)
├── rate_limit.py         Sliding-window per-IP RateLimiter + middleware
└── routers/
    ├── transactions.py   /transactions endpoints
    └── accounts.py       /accounts/:id/balance and /summary endpoints
tests/                    pytest suite (32 tests) covering all endpoints
demo/                     run script, sample requests, sample data
```

### Key design decisions

- **Type-aware balance/summary**: a `deposit` only credits `toAccount`, a `withdrawal` only debits `fromAccount`, and a `transfer` does both. This matches conventional banking semantics — the "other side" of a deposit/withdrawal represents an external source/destination, not a tracked internal balance movement.
- **Multi-currency balances**: balance and summary are returned as a per-currency breakdown (e.g. `{"USD": 100.0, "EUR": 50.0}`) rather than assuming a single currency per account, since nothing in the model prevents an account from transacting in multiple currencies.
- **Validation bypasses FastAPI's default 422 handler**: the request body is parsed manually and validated field-by-field so error responses always match the assignment's required shape (`error` + `details[]`) rather than FastAPI's default `422` format. A global handler still normalizes any other `RequestValidationError` (e.g. bad query params) into the same shape.
- **Server-generated fields**: `id`, `timestamp`, and `status` are always generated by the server and ignored/overwritten if sent by the client — clients only submit `fromAccount`, `toAccount`, `amount`, `currency`, `type`.
- **Unknown accounts are not 404s**: an account only exists implicitly via its transactions, so balance/summary for an account with no transactions returns zeroed-out results (`{}` balances, `0` count) rather than a 404.
